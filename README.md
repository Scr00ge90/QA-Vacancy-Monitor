# 🤖 Job Monitor v2.0

> Автоматизированная система поиска вакансий и отправки откликов для Junior QA / стажёров  
> **Python · Telegram API · Selenium · FastAPI · Web UI**

---

## 📌 О проекте

Поиск первой работы в QA — это десятки каналов в Telegram и сотни вакансий на HH.ru. Вместо ручного мониторинга я автоматизировал весь процесс: система работает в фоне, находит подходящие вакансии и отправляет отклики быстрее, чем успеваешь прочитать объявление.

Проект начинался как простой Python-скрипт с bat-меню, вырос до полноценного веб-приложения с двумя модулями автоматизации, REST API и красивым интерфейсом.

---

## 🚀 Функциональность

### Telegram Monitor
- Мониторинг публичных каналов в реальном времени через **Telethon MTProto API**
- Умная фильтрация по ключевым словам и исключениям (junior/intern/стажировка vs senior/lead)
- Автоматическая отправка сообщения + PDF резюме найденным контактам
- Защита от дублей через постоянный файл `all_sent_users.txt`
- Лимит откликов в день (защита от бана)
- `SAFE_MODE` — тестовый режим без реальной отправки
- Парсинг истории каналов при старте
- Обработка `FloodWaitError` с автопаузой

### HH.ru Monitor
- Автоматический поиск вакансий через **Selenium + Chrome**
- Фильтры: регион, опыт, зарплата, график, период публикации
- Автоматический отклик с сопроводительным письмом
- Сохранение сессии через cookies (повторный вход не нужен)
- Редактор сценария автокликания прямо в UI

### Web Interface
- Единый дашборд для TG и HH с метриками
- Управление каналами, ключевыми словами, шаблонами
- Просмотр переписки с контактами прямо в браузере (Telegram чаты)
- Раздельные логи TG и HH с цветовой подсветкой
- Автозапуск скриптов при старте приложения

## ⚙️ Установка и запуск

### 1. Клонировать репозиторий
```bash
git clone https://github.com/Scr00ge90/QA-Vacancy-Monitor.git
cd QA-Vacancy-Monitor
```

### 2. Установить зависимости
```bash
pip install -r requirements.txt
```

### 3. Настроить окружение
```bash
cp .env.example .env
```

Заполнить `.env`:
```
TG_API_ID=ВАШ_ID        # с https://my.telegram.org/apps
TG_API_HASH=ВАШ_HASH
SAFE_MODE=true           # true = тест, false = боевой режим
```

### 4. Запустить

**Windows** — двойной клик:
```
start_web.bat
```

**Linux/Mac:**
```bash
uvicorn api.main:app --host 127.0.0.1 --port 8000
```

Откроется браузер на `http://127.0.0.1:8000`

## 🧪 Тестирование и найденные баги

В процессе разработки я применял QA-подход к собственному проекту:

### Критические баги (найдены и исправлены)

| # | Баг | Причина | Фикс |
|---|---|---|---|
| 1 | Повторная отправка сообщений после перезапуска | Дневной лог сбрасывался, `sent_users` очищался | Добавлен постоянный `all_sent_users.txt` |
| 2 | Превышение лимита (35 вместо 25) | Лимит проверялся до цикла, но не внутри при multiple usernames | Проверка добавлена внутрь цикла по username |
| 3 | Конфликт Telegram сессий | `monitor.py` и `app.py` использовали одну `session` | Разделены на `session` и `session_web` |
| 4 | `Too many requests` без паузы | `FloodWaitError` игнорировался, скрипт продолжал слать | Добавлена обработка с `e.seconds` паузой |
| 5 | `ModuleNotFoundError: api` | uvicorn запускался не из директории проекта | Добавлен `cd /d "%BASE%"` в батник |
| 6 | JS `SyntaxError` из-за двойного экранирования | Regex в JS патчился через Python `str.replace` | Пересборка файла с нуля |
| 7 | Дублирование каналов в конфиге | Не было проверки при добавлении | Добавлена валидация на дубли |
| 8 | PID-файл не удалялся после краша | `atexit` не срабатывал при `SIGKILL` | `is_running()` чистит мёртвый PID-файл |

### Логические проблемы (найдены и исправлены)

- `api_hash` сохранялся в `config.json` открытым текстом → вынесен только в `.env`
- `load_config()` вызывался в цикле при каждом сообщении → кеш с TTL 30 сек
- `get_sent_list()` вызывал `load_config()` на каждую строку лога → вынесен за цикл
- Хардкод фильтра `["junior", "стаж", "intern"]` дублировал `keywords` из конфига → убран
- `system.log` рос бесконечно → добавлена ротация (5000 строк)
- CORS `allow_origins=["*"]` → ограничен до localhost

### Тест-кейсы (SAFE_MODE)

```
TC-001: Скрипт находит вакансию с ключевым словом "junior"
TC-002: Скрипт не откликается на вакансию с "senior" в тексте  
TC-003: Скрипт пропускает @bot аккаунты
TC-004: Повторная отправка одному контакту исключена
TC-005: При достижении лимита скрипт останавливается
TC-006: При FloodWaitError скрипт делает паузу и продолжает
TC-007: После перезапуска не отправляет повторно
TC-008: Настройки из UI применяются без редактирования кода
```
